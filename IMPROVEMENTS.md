# AKFES v2 개선 방안

현재 v2 브랜치는 라이선스 인증, 장치 바인딩, Arduino 키패드 입력, 요청 서명, AES-256-GCM 파일 처리, 네이티브 저장까지 하나의 흐름으로 연결되어 있습니다. 다음 개선은 안정성·대용량 처리·운영 보안을 우선합니다.

## 1. 대용량 파일 처리

현재 파일은 JSON Base64로 전송되므로 원본보다 전송량이 약 33% 증가하고 클라이언트와 서버 메모리에 파일 복사본이 여러 개 생깁니다.

개선 방향:

- Tauri가 선택한 파일을 임시 경로로 관리
- `multipart/form-data` 또는 바이너리 스트림 API 도입
- 서버에서 청크 단위 암호화·복호화
- 임시 파일을 원자적으로 결과 파일로 교체
- 업로드·처리 진행률과 취소 기능 제공
- 중단된 임시 파일 자동 정리

## 2. 네이티브 파일 처리 강화

이번 단계에서 Rust 네이티브 저장 대화상자와 직접 파일 쓰기를 추가했습니다.

추가 개선 방향:

- 파일 선택도 WebView `<input>` 대신 네이티브 열기 대화상자로 변경
- 저장 취소·저장 성공 경로를 React 상태에 명확히 표시
- 기존 파일 덮어쓰기 전 확인
- 저장 후 탐색기에서 위치 열기 기능
- 출력 파일 권한과 심볼릭 링크 처리 검증

## 3. 세션과 인증

- 서버 명시적 로그아웃 API 추가
- 로그아웃 시 세션과 남은 챌린지 즉시 폐기
- 연속 로그인 실패 횟수 제한과 지수 백오프
- 관리자 API에 역할별 권한과 별도 관리자 세션 도입
- 감사 로그 위변조 탐지를 위한 해시 체인 또는 외부 로그 저장소 연동
- 운영 환경에서는 HTTPS만 허용

## 4. 장치 바인딩

현재 장치 ID는 클라이언트가 생성한 UUID 기반입니다. 이는 복사될 수 있으므로 강한 하드웨어 신원은 아닙니다.

개선 방향:

- Windows TPM 또는 OS 보안 저장소에서 장치 키쌍 생성
- 서버 챌린지를 장치 개인키로 서명
- 개인키는 내보낼 수 없도록 저장
- 관리자 승인 기반 장치 교체 절차
- 장치별 마지막 접속·버전·폐기 상태 관리

## 5. 키패드와 비밀번호

- 비밀번호 입력 완료 후 메모리에서 가능한 빨리 제거
- React 문자열 대신 Rust의 민감정보 컨테이너 사용 검토
- 입력 횟수·길이 제한과 잠금 정책
- 키패드 통신에 세션 nonce 또는 간단한 무결성 검증 추가
- 포트 자동 선택 시 USB VID/PID와 펌웨어 식별값 검증

## 6. 암호화 포맷

- 대용량 스트리밍용 청크 포맷 버전 추가
- 각 청크에 순번과 인증 태그 포함
- 파일 전체 메타데이터에 최종 무결성 값 포함
- 포맷 파서 퍼징과 손상 파일 테스트 확대
- 암호화 정책 버전과 알고리즘 교체 절차 문서화

## 7. 테스트와 CI

- GitHub Actions에 Python pytest·Ruff 추가
- TypeScript 빌드·ESLint 또는 Biome 검사
- Rust `cargo fmt`, `cargo clippy`, `cargo test`
- Windows Tauri 빌드 검사
- 파일 암호화 왕복 테스트를 실제 클라이언트까지 포함
- 잘못된 서명·재사용 챌린지·만료 세션 통합 테스트
- 0바이트·한글 파일명·매우 긴 파일명·대용량 파일 테스트

## 8. 배포와 운영

- Windows 아이콘과 MSI/NSIS 설치 프로그램 구성
- 코드 서명 인증서 적용
- 서버 주소를 빌드 채널별로 분리
- 자동 업데이트 서명 검증
- 개발용 기본 비밀값이 포함된 빌드 차단
- 서버 데이터베이스 백업·복구와 키 교체 절차 마련

## 권장 우선순위

1. GitHub Actions CI와 전체 빌드 오류 제거
2. 네이티브 파일 선택 및 저장 상태 피드백
3. 서버 로그아웃과 세션 즉시 폐기
4. 대용량 파일 스트리밍
5. TPM 기반 장치 키
6. Windows 설치 프로그램·코드 서명·자동 업데이트
